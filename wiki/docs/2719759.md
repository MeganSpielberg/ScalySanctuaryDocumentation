# Devlog: RPG Style Dialogue System

> ℹ️ **Note:** Author: Valentino
> Analysis & Design:
> [RPG Style Dialogue System - Analysis and Design.pdf](images/2719759/3571725.pdf)

## 🤔 Challenge

***How can a flexible dialogue system be structured to handle both
linear tutorial guidance and dynamic state-based feedback for crafting
outcomes?***

The primary goal of this devlog was to develop a "Hardcoded" dialogue
system that serves as a reliable knowledge resource for players within
the feeding module. Unlike a standard linear narrative, this system
needed to bridge the gap between static text and interactive gameplay,
responding to the player's actions in real-time.

## 💡 Methods

### 📚 Library: Technical Design Study

I researched common dialogue system architectures within the Godot
ecosystem, ultimately settling on a **Dictionary-based node system**. By
structuring dialogue as a nested tree, I ensured that each node
contained not only text but also character metadata (moods/textures) and
an array of player options that point to subsequent node IDs.

The full analysis and design can be found by following the evidence
description.

### 🛠️ Workshop: Prototyping

The implementation was split into two specialized components to maintain
a "Separation of Concerns":

- **The Dialogue Manager
  (**[<strong>dialoguemanager.gd</strong>](http://dialoguemanager.gd/)**):** A UI
  controller handling visual rendering, typewriter tweens, and character
  state.

- **The Trigger Bridge
  (**[<strong>VetCraftingTrigger.gd</strong>](http://vetcraftingtrigger.gd/)**):** An
  observer that monitors the
  [Bowl.gd](http://bowl.gd/) script and connects gameplay events (like
  adding an ingredient or failing a recipe) to specific narrative nodes.

### ✅ Lab: Validation

Validation was done through play testing and observing how testers
interacted with the dialogue system. More on those user tests can be
found in the remainder of this devlog.

## 🎨 Design & Functionality

### 🌳 The Dialogue Tree Architecture

The system stores narrative data in a dialogue_tree dictionary. Each
entry represents a "Node" with specific properties. This structure
allows for complex branching logic where player choices dictate the next
state.

#### 💻 GDScript Example

```gdscript
var dialogue_tree = {
    "node_L1_dishfail": {
        "text": "Remember, Leopard Geckos are strict insectivores! Fruit like bananas can make them very sick.",
        "options": [
            {"text": "Why no fruit?", "next": "node_explanation_fruit"},
            {"text": "I'll try again.", "next": "back"}
        ]
    }
}
```

### ⌨️ The Typewriter & Visual Feedback Loop

To simulate a living conversation, I utilized Godot’s Tween system. The
*on*typewriter_step function calculates the typing duration based on
string length, ensuring the text doesn't appear too fast or too slow.

- **Character States:** The system references a vet_textures dictionary.
  When a node is loaded, the manager can swap the character's sprite to
  **Neutral**, **Talking**, or **Worried**.

- **Audio Sync:** A type_sound (AudioStreamPlayer) is triggered in the
  \_process loop or via tween callbacks to provide a tactile feel to the
  text.

#### 💻 GDScript Example

```gdscript
func ontypewriter_step(data: Dictionary) -> void:
    response_label.visible_characters = 0
    vet_character.texture = vet_textures["Vet Talking"] # Animation state
   
    var duration = data.text.length() / typing_speed
    typing_tween = create_tween()
    typing_tween.tween_property(response_label, "visible_characters", data.text.length(), duration)
   
    # Reveal UI buttons only after the text has finished typing
    typing_tween.finished.connect(_show_options.bind(data.options))
```

### 🔗 Contextual Event Handling

The [VetCraftingTrigger.gd](http://vetcraftingtrigger.gd/) is the "brain" that connects
the bowl to the vet. It uses an *process loop to watch the
bowl.ingredient*nodes.size().

- **Intro Triggers:** When the first ingredient is added, it checks
  GameFlowManager.get_loop_count() to determine which "Intro" node to
  play.

- **Outcome Triggers:** It connects to the on_dish_fail signal from the
  Bowl. By using await get_tree().create_timer(0.4).timeout, it ensures
  the player sees the physical ingredients slide back to their start
  positions before the NPC interrupts with corrective feedback.

## ✅ Validation

### 🔍 Key Insight: The Vet as a Primary Guide

The most significant finding from Playtest R10 was that players in the
RPG condition interacted with the Vet significantly more than those in
the Basic condition, who consulted the clipboard**.** For many testers,
the Vet became the primary interface for understanding the game's "why"
and "how."

- **Instructional Reliance (Tester 11):** Tester 11 (Game Production
  Student) initially felt confused by the feeding mechanics but engaged
  with the Vet immediately upon his appearance. They "speedran" the
  dialogue but reported understanding the mixing and variety mechanics
  quickly *because* of that interaction.

- **Scaffolding for Novices (Tester 8):** Tester 8 (a younger player)
  was unable to succeed at the crafting task through trial and error
  alone. They only achieved the correct recipe after being prompted by
  the system to ask the Vet for the specific recipe requirements. This
  confirmed that the dialogue system acts as a necessary "scaffold" for
  players with less prior knowledge.

- **Independent Discovery (Tester 2):** In contrast, some players used
  the Vet as a secondary check. Tester 2 discovered the correct recipe
  on Day 2 through experimentation but then consulted the Vet to confirm
  *why* their choice worked, demonstrating the system's value for
  reinforcing already-acquired knowledge.

### ✨ Observations on System "Juice"

The RPG aesthetics (typewriter effect and portrait swapping) were noted
as positive engagement factors. Tester 11 specifically mentioned that
the system felt "pretty well built" and "kinda fun," suggesting that the
RPG-style presentation made the educational content more palatable than
a static text file.
