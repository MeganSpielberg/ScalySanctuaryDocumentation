## 🤔 Challenge

***How can a flexible, recipe-based crafting system be built in Godot to
handle dynamic ingredient validation and state-based feedback?***

The technical goal for this module was to develop a robust system
capable of processing drag-and-drop interactions where multiple items
(ingredients) are combined to produce a specific outcome (a dish). The
architecture needed to support alphabetical, order-independent
validation so players weren't penalized for the sequence of their
actions. Additionally, the script had to be condition-agnostic, allowing
it to communicate with either an LLM-driven NPC manager or a static
dialogue system, depending on the active experimental variable.

### 🎨 Design

#### 💻 System Design

The system is built around a centralized
[Bowl.gd](http://bowl.gd/) script that manages a list of active
ingredient nodes and compares them against a dictionary of defined
recipes.

#### 🏗 Architecture

The architecture relies on [Godot](Godot_851969.md)'s signal system to communicate results
to a vet_manager node, which handles the narrative response.

| Script | Responsibility |
|----|----|
| [<strong>Bowl.gd</strong>](http://bowl.gd/) | Tracks ingredients, handles Area2D signals, performs recipe sorting/matching, and manages UI buttons. |
| [<strong>Ingredient.gd</strong>](http://ingredient.gd/) | Manages local state (e.g., over_bowl), handles the "return-to-home" tweening logic on failure, and stores its own name/color data. |
| **GameFlowManager** | A singleton that injects the current active_recipe into the bowl at runtime. |

#### ✅ Recipe Validation Logic

The core of the logic resides in the *on*cook_button_pressed function.
By using .sort() on both arrays, we eliminate the need for complex
nested loops or positional tracking:

```gdscript
func on_cook_button_pressed():
    var current_names = []
    for node in ingredient_nodes:
        current_names.append(node.ingredient_name) # [cite: 58]

    current_names.sort() # Alphabetical sorting for order-independence
    var required = RECIPES[active_recipe].duplicate()
    required.sort()

    if current_names == required: # Simple array comparison [cite: 58]
        finish_dish()
    else:
        fail_dish()
```

### 📊 Results

#### 🔨 Implementation

- **Ingredient Management:** Ingredients are tracked via
  *on*area_entered and *on*area_exited. Once three ingredients are
  added, the "Make Meal" button is enabled via code.

- **Failure Recovery:** On a failed recipe check, the script iterates
  through ingredient_nodes and calls node.reset_fail(), which triggers a
  programmed slide back to the ingredient's original spawn point.

#### 🔍 Validation

Playtesting provided critical insights into the player's mental model
and visual feedback loop:

**Visual State Confusion:** A "full bowl" bug (Tester 3) and confusing
color-blending logic (multiple testers) were identified. Players could
not reliably interpret what mixed colors signaled, often assuming the
meal was ready regardless of the count. Consequently,
show_filled_visual() has been temporarily commented out for a redesign.

The full playtest results can be found by following the evidence
description.
